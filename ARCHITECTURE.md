# System Architecture - Chatbot Platform

This document describes the high-level architecture, flow diagrams, data model design, and core design choices implemented in the Chatbot Platform application.

---

## 1. System Overview

The Chatbot Platform is structured as a decoupled Single Page Application (React/TypeScript client) communicating over standard JSON and SSE (Server-Sent Events) protocols with an asynchronous API server (FastAPI/Python/PostgreSQL).

```
┌─────────────────┐           HTTP API (JSON) / SSE           ┌─────────────────┐
│                 │ ────────────────────────────────────────> │                 │
│  React Client   │                                           │   FastAPI App   │
│                 │ <──────────────────────────────────────── │                 │
└─────────────────┘              Token & Stream               └─────────────────┘
                                                                       │
                                                   ┌───────────────────┴───────────────────┐
                                                   ▼                                       ▼
                                         ┌───────────────────┐                   ┌───────────────────┐
                                         │  OpenAI Files API │                   │  PostgreSQL DB    │
                                         │  (Document Store) │                   │  (Metadata Store) │
                                         └───────────────────┘                   └───────────────────┘
```

## 2. Authentication Flow

Authentication is managed via JSON Web Tokens (JWT) using a short-lived Access Token in headers and a rotated, HTTP-Only, Secure, SameSite=Lax cookie-based Refresh Token in cookies.

### Register / Login
```
Client                      FastAPI (Auth Services)               PostgreSQL (DB)
  │                                   │                                  │
  │─── POST /login or /register ─────>│                                  │
  │    (Plain Credentials)            │─── Verify or Insert User ───────>│
  │                                   │<── Return User Record ───────────│
  │                                   │                                  │
  │                                   │─── Generate Token Pair ──────────│
  │                                   │─── Persist Refresh Hash ────────>│
  │                                   │                                  │
  │<── Success Response ──────────────│                                  │
  │    Header: Set-Cookie (Refresh)   │                                  │
  │    Body: { Access Token, User }   │                                  │
```

### Refresh Token Rotation (On 401 Unauthorized)
```
Client                      Axios Interceptor                FastAPI (Auth Services)
  │                                 │                                   │
  │─── Request with Access Token ──>│                                   │
  │    (Token Expired)              │─── POST /auth/refresh ───────────>│
  │                                 │    (Sends HttpOnly Cookie)        │
  │                                 │                                   │
  │                                 │                                   │── Verify expiry in DB
  │                                 │                                   │── Delete old token
  │                                 │                                   │── Generate new pair
  │                                 │                                   │── Save new token hash
  │                                 │                                   │
  │                                 │<── Success Response ──────────────│
  │                                 │    Cookie: Set-Cookie (New)       │
  │                                 │    Body: { New Access Token }     │
  │                                 │                                   │
  │<── Retry original request ──────│                                   │
```

## 3. Chat Streaming Flow

Streaming outputs prompt tokens generated by OpenRouter in real time back to the client using SSE.

```
React UI                     FastAPI Controller               OpenRouter API
  │                                  │                               │
  │─── POST /projects/{id}/chat ────>│                               │
  │    { message }                   │                               │
  │                                  │── Write User Msg in DB        │
  │                                  │── Read History Logs           │
  │                                  │                               │
  │                                  │── POST /chat/completions ────>│
  │                                  │    { stream: true }           │
  │                                  │                               │
  │                                  │<── Stream SSE Lines ──────────│
  │                                  │    data: {"choices": [...]}   │
  │<── Pipes Token SSE Chunk ────────│                               │
  │    data: {"token": "hello"}      │                               │
  │                                  │                               │
  │<── Ends Stream (data: [DONE]) ───│                               │
  │                                  │                               │
  │                                  │── Background Task:            │
  │                                  │    Persist Assistant Msg in DB│
```

## 4. Database Design Rationale

The database tables are designed to support relational consistency while keeping queries lightweight.

- **`users`**: Contains credential hashes. Indexed on `email` to ensure fast login lookup.
- **`refresh_tokens`**: Maintains cryptographically secure SHA-256 hashes of generated refresh tokens. It references `users.id` with a cascading delete constraint, meaning when a user profile is deleted, all their associated tokens are automatically cleared.
- **`projects`**: Configures individual workspace configurations. It includes a foreign key constraint linking to the owner user, preventing cross-tenant access.
- **`messages`**: Dialogue history logs. Large conversational lists are paginated on retrieval.
- **`project_files`**: Matches uploaded knowledge base documents with OpenAI file IDs.

## 5. Technology Selection Rationale

### Why FastAPI?
- **Async-First Execution**: Built on Starlette, it handles concurrent network operations (like streaming SSE or uploading files to OpenAI) with high efficiency, matching the requirements of scalable applications.
- **Pydantic Validation**: Restricts request models on the borders of endpoints before any controller or database transactions are hit.
- **Documentation**: Automatically structures Swagger JSON UI directories out-of-the-box.

### Why Zustand?
- **Simplicity**: Lighter and less boilerplate than Redux, Zustand makes global state accessible in simple hooks.
- **Persist Middleware**: Automatically stores authenticated token structures in client `localStorage` with minimal configuration.
